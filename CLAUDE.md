# CLAUDE.md — 手取りのホント(仮) プロジェクト規約

このファイルは、このリポジトリで作業する Claude Code に向けた恒久的な指示です。
セッションが変わっても、この内容は常に有効です。

## 0. 最初に必ずやること

作業に着手する前に、必ず `SPEC.md` を読むこと。SPEC.md はこのサイトの仕様の唯一の正とする文書であり、要件・ページ構成・入力項目・スコープ外・完了条件はすべてここに定義されている。SPEC.md を読まずにコードを書き始めてはならない。

会話の最初の指示が実装依頼であっても、まず SPEC.md を読み、その内容を踏まえた**実装計画**を提示し、ユーザーの承認を得てから着手すること。計画なしにファイルを作成・編集し始めてはならない。

## 1. 進行ルール(この順序を厳守する)

1. SPEC.md を読む
2. 着手する範囲(今回のセッションで何をどこまでやるか)を明確にする
3. 実装計画を提示する(変更するファイル、追加するファイル、処理の流れ、確認したい前提を含める)
4. ユーザーの承認を得る
5. 承認された計画どおりに実装する
6. 実装後、変更点を要約して報告する

推測で仕様を補わない。SPEC.md に書かれていないことで実装上の判断が必要な場合は、ユーザーに確認する。

## 2. スコープ厳守

SPEC.md の「6. スコープ外」に明記されている項目は実装しない。良かれと思って先回りで作らないこと(例: 保育料計算、副業・譲渡所得の反映、経費率スライダーをかんたん入力に混ぜる、など)。

「ついでにこれも」という提案自体はしてよいが、実装は必ずユーザーの承認後にする。

## 3. 変更要望が出た場合

実装中や実装後に、SPEC.md に書かれていない変更・追加の要望が出た場合、いきなりコードに反映しない。次の順序を踏む。

1. SPEC.md の該当箇所をどう変更するかの差分案を提示する
2. ユーザーの確認を得る
3. SPEC.md を更新する(バージョン番号と変更履歴を必ず更新)
4. 更新後の SPEC.md に基づいて実装する

SPEC.md の「変更履歴」テーブルは、行を追加する形で更新し、過去の版の記述を書き換えたり消したりしない。

## 4. リリース方針(同時設計・段階リリース)

このプロジェクトは「かんたん入力(第1弾)」「詳細入力(第2弾)」の2段階リリースを前提にしている。

- 計算エンジン・データ構造・型定義は、最初から詳細入力モードの項目(iDeCo、生命保険料、医療費、住宅ローン控除、経費率、青色申告、ボーナス比率など)を見据えて設計すること。あとから詳細モードを継ぎ足すのではなく、最初から両対応の構造にする。
- ただし UI 上の公開は段階的。第1弾では詳細入力タブは「準備中」として非表示またはグレーアウトし、かんたん入力のみで完結させる。
- 「今回は第1弾のどこまでを実装するか」は、その都度の実装計画で明示すること。

## 5. データと計算に関する注意

- 国保料率・税率・控除額などの数値は、必ず SPEC.md に記載された一次ソース(国税庁、厚労省、都道府県公表資料など)に基づくこと。数値を推測・仮定で埋めない。
- 未整備の市区町村データは、平均値や近隣自治体の値で代替しない。「データ未整備」として明示する(SPEC.md 5章の方針どおり)。
- 計算結果には常に「概算である」旨、出典、対象年度を添えること。
- YMYL(税務・お金)領域であることを常に意識し、断定的な表現、個別の税務相談に該当しうる表現を避ける。

## 6. Git 運用

- コミットメッセージは `git commit -F` で作成する(ヒアドキュメントでの直接指定は避ける)
- push 前に必ず `git fetch` を行い、origin/main との差分を確認する
- 生成ファイル(市区町村別ページなど自動生成物)がコンフリクトした場合: origin/main の生成ファイルを一旦ベースにし、生成スクリプトを再実行してから、生成ファイルには `ours` 戦略でマージし、手動のコード変更は個別に保持する
- 作業ツリーに未コミットの WIP がある状態で別件の変更を扱う場合は、`git apply --check` で影響範囲を確認してから `git stash` でパッチ分離する

## 7. 完了報告の基準

「完了しました」は次の3点をすべて確認してから報告すること。1つでも欠けた状態で完了と報告しない。

1. push が成功している
2. GitHub Actions のワークフローが成功している
3. 本番の GitHub Pages で実際に変更が反映されていることを確認している

## 8. 権限について

破壊的な操作(force push、履歴の書き換え、生成データの一括削除など)は、実行前に必ずユーザーに確認する。読み取り専用の調査(ファイル閲覧、grep、ビルド確認)は都度の確認なしに進めてよい。
